# Paper Analysis: DS-STAR & DS-STAR+

**Paper**: "DS-STAR: Data Science Agent for Solving Diverse Tasks across Heterogeneous Formats and Open-Ended Queries"  
**Authors**: Jaehyun Nam (KAIST), Jinsung Yoon, Jiefeng Chen, Raj Sinha, Tomas Pfister (Google Cloud)  
**Published**: arxiv:2509.21825 — September 2025 (v4: February 2026)  
**Date Analysed**: 2026-06-24  

---

## Overview

DS-STAR is a multi-agent data science system that can autonomously answer questions over heterogeneous, multi-file data without requiring the user to write any code. It is composed of two systems built on the same inner engine:

- **DS-STAR**: Answers specific, factoid, or multi-step analytical questions
- **DS-STAR+**: Generates comprehensive, cited research reports for open-ended queries

The core insight is that a single large language model asked to "analyse this data and answer my question" performs poorly, while a structured multi-agent pipeline — where each agent does one thing and passes its output to the next — achieves state-of-the-art performance.

## Part 1: DS-STAR — Core Architecture

### 1.1 Formal Notation

| Symbol | Meaning |
|--------|---------|
| `q` | User query in natural language |
| `D` | File descriptions — textual summaries generated by the Analyzer for all data files |
| `p₀..pₖ` | Cumulative plan — a list of step descriptions, one step added per round |
| `sₖ` | Python script generated in round k, implementing the full cumulative plan |
| `rₖ` | Execution result (stdout) from running `sₖ` in the sandboxed environment |
| `k` | Current round index, starting at 0; maximum 20 rounds |

### 1.2 The Eight Agents

#### Agent 1: Analyzer

**What it does**: Runs before anything else. Examines every file in the data directory and produces a natural-language description of each.

**Supported file formats**: CSV (auto-detects comma / semicolon / tab delimiters), JSON, Markdown, Excel (.xlsx), unstructured text

**Output (`D`)**: A per-file description containing:
- Column names and data types
- Number of rows and columns
- First 5 sample rows
- Null/missing value counts
- Unique value counts for categorical columns
- Statistical summary (min, max, mean, std) for numeric columns

**Mechanism**: The Analyzer itself generates a Python profiling script, runs it inside the Docker sandbox, captures the stdout, and returns the text descriptions to the state.

**Why it is critical**: Ablation study in the paper shows that removing the Analyzer alone drops accuracy on hard tasks from **45.2% → 26.98%** (a drop of 18 percentage points). It is the single most impactful component in the entire system. Without it, the Planner has no grounding in what data is actually available and generates incorrect plans.

---

#### Agent 2: Planner

**What it does**: Generates exactly one new step per round. Plans are built incrementally, one step at a time.

**Round 0 — Initialisation**:
- Receives: `(q, D)`
- Produces: `p₀` — the simplest possible first step (e.g., "Load the fraud loss table and compute basic summary statistics by LFI")

**Round k — Continuation**:
- Receives: `(q, D, p₀..pₖ₋₁, rₖ₋₁)` — the full history plus the last execution result
- Produces: `pₖ` — the next step, explicitly conditioned on what was observed in `rₖ₋₁`
- This is the key design: the Planner responds to what actually happened, not what was predicted

**Why one step at a time**: Committing to a full upfront multi-step plan is brittle. If step 3 depends on output from step 2 that turns out to be structured differently than expected, the entire plan becomes invalid. One-step planning mirrors how a human works in a Jupyter notebook: observe, decide next action, observe again.

**Output**: Appends `pₖ` to the cumulative plan state: `{p₀, p₁, ..., pₖ}`

---

#### Agent 3: Coder

**What it does**: Converts the current cumulative plan into an executable Python script.

**Round 0**: Given `(q, D, p₀)` → produces `s₀` (initial script)

**Round k**: Given `(q, D, p₀..pₖ, sₖ₋₁)` → extends the prior script with the new step

**Hard constraints enforced in the prompt**:
- Always use full absolute paths: `/workspace/data/{filename}`
- Memory-safe loading: use `nrows=10000` for large CSV files
- Pure Python only — no shell commands, no subprocess calls
- For CSV files: try comma delimiter first, then semicolon, then tab
- All output must be printed to stdout (the Executor captures stdout as `rₖ`)

**Output**: `sₖ` — the complete Python script implementing all steps accumulated so far. The script is a full, runnable file, not just the new step in isolation.

---

#### Agent 4: Executor

**What it does**: Runs `sₖ` inside an isolated Docker container and captures the result.

**Sandbox configuration**:
- `--network=none`: complete network isolation, no internet access
- `--memory=2g`: 2 GB memory cap to prevent runaway processes
- Data directory mounted read-only: the script can read data but cannot modify it
- Non-root user inside the container: principle of least privilege
- Per-execution timeout: enforced to prevent infinite loops

**Output**:
- `rₖ`: stdout from the script execution (capped at a character limit)
- `stderr`: error output, capped
- `exit_code`: 0 for success, non-zero for any failure

---

#### Agent 5: Debugger

**What it does**: Activated only when `exit_code != 0`. Attempts to repair the failing script.

**Receives**: `(sₖ, stderr, traceback)`

**Produces**: A fixed version of `sₖ` — the Debugger repairs the code but **never modifies the plan**. It does not create a new step; it only fixes the implementation of the current step.

**Key distinction**:
- Debugger fixes **code errors** (wrong column name, wrong pandas API, type mismatch)
- Router fixes **plan errors** (wrong analytical direction, incorrect logic)

**Limit**: Maximum 3 debug attempts per round. If the script still fails after 3 attempts, the best-effort result is passed to the Verifier anyway.

---

#### Agent 6: Verifier

**What it does**: Acts as an LLM-based judge. Determines whether the current result adequately answers the original query.

**Receives**: `(q, p₀..pₖ, sₖ, rₖ)` — the full accumulated context

**Returns**: Binary verdict — `"sufficient"` or `"insufficient"`

**What it actually checks**: Not just "did the code run without errors?" but "does the actual output `rₖ` answer the user's question `q`?" The Verifier is grounded in the execution result, not just the plan.

**Routing based on verdict**:

| Condition | Next Agent |
|-----------|-----------|
| `"sufficient"` | Finalizer |
| `"insufficient"` + `k < max_rounds` | Router |
| `"insufficient"` + `k == max_rounds` | Finalizer (best-effort output) |

---

#### Agent 7: Router

**What it does**: When the plan is insufficient and rounds remain, decides HOW to improve the plan.

**Receives**: `(q, D, p₀..pₖ, rₖ)`

**Returns**: Either the token `"Add Step"` or an integer index `l ∈ {1, ..., k}`

**"Add Step" decision**:
- The overall plan direction is correct, but it is incomplete
- The system retains all existing steps and the Planner adds `pₖ₊₁`
- Example: "We loaded the data and computed totals, but haven't yet computed the peer percentile rank"

**Index `l` — Backtrack decision**:
- Step `l` is identified as the root of the error
- The system **truncates** the plan to `{p₀, ..., pₗ₋₁}` — steps `pₗ` onwards are discarded
- The Planner then regenerates `pₗ` fresh
- Example: "Step 2 joined on the wrong column. Regenerate from step 2."

**Why truncate-and-regenerate instead of direct correction**: The system deliberately does not ask the LLM to directly fix the erroneous step. Truncation forces the Planner to regenerate with its random sampling temperature, producing a different (and hopefully correct) plan. This avoids the LLM producing a cosmetically-modified version of the same incorrect step.

**Ablation result**: Removing the Router (making the system add steps sequentially only, with no backtracking) measurably reduces performance. Backtracking is essential for correcting logical errors in earlier steps.

---

#### Agent 8: Finalizer (called "Finalyzer" in the paper)

**What it does**: Takes the last successful code and result, applies the user's formatting preferences, and produces the final output delivered to the user.

**Receives**: `(sₖ, rₖ, formatting_guidelines)`

**`formatting_guidelines`**: User-specified preferences, e.g.:
- "Round all monetary values to 2 decimal places"
- "Output the result as a JSON object"
- "Summarise the findings in 3 bullet points"
- "Format as a table with columns: LFI Name, Fraud Loss AED, Peer Rank"

**Mechanism**: The Finalizer may generate and run a short additional script to reformat the output according to the guidelines, rather than simply applying text transformations.

**Output**: The final cleaned, formatted answer ready for the user.

### 1.3 Complete DS-STAR Pipeline

```
DATA FILES IN DIRECTORY
         │
         ▼
[ANALYZER]  ─────────────────────────────────────  D = {file₁: desc₁, file₂: desc₂, ...}
         │
         ▼
[PLANNER]  Round 0: p₀ = first step from (q, D)
         │
         ▼
[CODER]  s₀ = Python script from (q, D, p₀)
         │
         ▼
[EXECUTOR]  r₀ = run s₀ in Docker sandbox
         │
         ├── exit_code ≠ 0 ──► [DEBUGGER] ──► [EXECUTOR]  ←── max 3 attempts
         │
         ▼
[VERIFIER]  receives (q, p, s, r)
         │
         ├── "sufficient"  ────────────────────────────────────────► [FINALIZER] → OUTPUT
         │
         ├── "insufficient" + k == 20  ───────────────────────────► [FINALIZER] → OUTPUT (best-effort)
         │
         └── "insufficient" + k < 20
                   │
                   ▼
               [ROUTER]
                   │
                   ├── "Add Step"
                   │       └──► [PLANNER] adds pₖ₊₁ ──► [CODER] ──► [EXECUTOR] ──► [VERIFIER] ↻
                   │
                   └── backtrack to step l
                           └──► truncate plan to {p₀..pₗ₋₁}
                                └──► [PLANNER] regenerates pₗ ──► [CODER] ──► [EXECUTOR] ──► [VERIFIER] ↻
```

## Part 2: DS-STAR+ — Research Report Extension

DS-STAR answers specific questions. DS-STAR+ answers **open-ended research queries** — queries that do not have a single numerical answer but instead require a structured report covering multiple angles of a topic.

DS-STAR is the **inner engine** of DS-STAR+. DS-STAR+ wraps it with a decomposition layer and a synthesis layer.

### 2.1 When DS-STAR+ Is Used vs DS-STAR

| Query Type | System | Example |
|------------|--------|---------|
| Specific factoid | DS-STAR | "How many STRs did Bank X file in Q3 2024?" |
| Comparative analysis | DS-STAR | "Which LFIs have the highest fraud loss rate vs. peer median?" |
| Multi-step investigation | DS-STAR | "Find customers with potential structuring patterns across Q1 2025" |
| Thematic research | DS-STAR+ | "Generate a thematic analysis of APP fraud growth across UAE banks in 2024" |
| Sector-wide trend report | DS-STAR+ | "Produce a comprehensive fraud trend report for CBUAE leadership covering H1 2025" |
| Regulatory briefing | DS-STAR+ | "Create an analytical annex for our supervisory letter to LFI Y" |

---

### 2.2 DS-STAR+ Components

#### Component A: Sub-Question Generator

**What it does**: Receives the open-ended query and decomposes it into a set of focused, answerable sub-questions that together cover the full scope of the original query.

**Input**: `q_open` — an open-ended research query

**Output**: `{sq₁, sq₂, ..., sqₙ}` — a list of focused sub-questions

**Example decomposition**:
- Input: "Generate a thematic analysis of APP fraud growth across UAE banks in 2024"
- Output:
  - `sq₁`: "What is the total APP fraud loss in AED by LFI for 2024 vs. 2023?"
  - `sq₂`: "Which channels (mobile banking, internet banking, telephone) have the highest APP fraud rates?"
  - `sq₃`: "Which LFIs saw the sharpest year-on-year increase in APP fraud losses?"
  - `sq₄`: "What is the peer median APP fraud detection rate, and which LFIs are below it?"
  - `sq₅`: "What proportion of APP fraud losses were recovered?"

**LLM assignment**: Gemini 2.5 Flash (sub-question generation is structurally straightforward and does not require the most capable model)

---

#### Component B: DS-STAR Inner Engine (per sub-question)

**What it does**: Each `sqᵢ` is passed independently through the full DS-STAR pipeline.

**Process**: For each `sqᵢ`:
1. Run Analyzer → Planner → Coder → Executor → Debugger → Verifier → Router ↻ → Finalizer
2. Produce: answer `aᵢ`, the Python code `sᵢ` that generated it, execution result `rᵢ`

**Output**: A structured evidence set `{(sq₁, a₁, s₁), (sq₂, a₂, s₂), ..., (sqₙ, aₙ, sₙ)}`

**LLM assignment**: Gemini 2.5 Flash for all inner DS-STAR agents (cost-efficient; applied n times in parallel or sequentially)

**Key property of the evidence set**: Each answer `aᵢ` is backed by runnable, auditable code `sᵢ`. Every claim in the final report can be traced back to the exact code that produced it.

---

#### Component C: Report Writer

**What it does**: Synthesises all sub-question answers into a single, structured, comprehensive research report.

**Receives**: `(q_open, {(sqᵢ, aᵢ, sᵢ)})` — the original open-ended query plus the full evidence set

**Produces**: A structured research report where every analytical claim is **explicitly cited** to the sub-question and code that produced it.

**Citation format**: *"APP fraud losses increased 34% year-on-year across the sector [sq₁, fraud_trend_by_lfi.py, line 23]"*

**Why citations matter**: In a regulatory environment, every claim in a supervisory report must be defensible. The citation trail — from conclusion back to sub-question back to Python code back to raw data — is the audit trail.

**LLM assignment**: Gemini 2.5 Pro — the most capable model is reserved exclusively for this synthesis step, where quality matters most. The Report Writer is explicitly instructed to cite, not to fabricate.

---

#### Component D: Evaluator

**What it does**: Reviews the generated report for completeness and depth relative to the original open-ended query. Decides whether the report is sufficient or needs more sub-questions.

**Receives**: `(q_open, report)`

**Returns**:
- `"sufficient"` → report proceeds to Explainer → user
- `"insufficient"` → triggers another round of the Sub-Question Generator with new or deeper sub-questions

**Refinement loop**: The paper finds a positive correlation between the maximum number of refinement rounds and report quality. More iterations generally produce better reports.

**LLM assignment**: Gemini 2.5 Flash

---

### 2.3 Complete DS-STAR+ Pipeline

```
OPEN-ENDED QUERY (q_open)
         │
         ▼
[SUB-QUESTION GENERATOR]  ─── Flash model
         │
         ▼
{sq₁, sq₂, ..., sqₙ}
         │
         ├──► DS-STAR(sq₁) → (sq₁, a₁, s₁)  ┐
         ├──► DS-STAR(sq₂) → (sq₂, a₂, s₂)  │  ← each runs the full 8-agent pipeline
         ├──► ...                              │     using Flash models
         └──► DS-STAR(sqₙ) → (sqₙ, aₙ, sₙ)  ┘
         │
         ▼
evidence set: {(sq₁,a₁,s₁), ..., (sqₙ,aₙ,sₙ)}
         │
         ▼
[REPORT WRITER]  ─── Pro model (highest quality reserved for synthesis)
         │  every claim cited to sqᵢ and sᵢ
         ▼
[EVALUATOR]  ─── Flash model
         │
         ├── "sufficient"  ──► final cited report → Explainer → PII Masker → Audit Log → user
         └── "insufficient" ──► [SUB-QUESTION GENERATOR] (new/deeper questions) ↻
```

### 2.4 LLM Model Assignment Summary

| Component | Model Tier | Reason |
|-----------|-----------|--------|
| Analyzer | Flash | Structured profiling; no complex reasoning needed |
| Planner | Flash | Each step is short; reasoning is guided by result |
| Coder | Flash | Code generation; correctness enforced by Executor |
| Debugger | Flash | Pattern-matching error fix |
| Verifier | Flash | Binary decision; well-structured input |
| Router | Flash | Constrained decision; "Add Step" or index |
| Finalizer | Flash | Formatting; constrained task |
| Sub-Question Generator | Flash | Decomposition; structurally simple |
| DS-STAR inner engine | Flash | Applied n times; cost efficiency matters |
| **Report Writer** | **Pro** | Open-ended synthesis; quality is paramount |
| Evaluator | Flash | Binary sufficiency judgment |

## Part 3: Benchmark Results

### DS-STAR (factoid / analytical queries)

| Benchmark | Task Type | Gemini 2.5 Pro alone | DS-STAR |
|-----------|-----------|---------------------|---------|
| DABStep | Overall | 41.0% | **45.2%** |
| DABStep | Hard tasks only | 12.70% | **45.24%** |
| KramaBench | Multi-file analysis | 39.8% | **44.7%** |
| DA-Code | Code-based QA | 37.0% | **38.5%** |

On hard DABStep tasks specifically, DS-STAR delivers a **+32 percentage point** improvement over using the same model alone. The system around the model matters more than the model itself on complex tasks.

### DS-STAR+ (open-ended research reports)

| Benchmark | Evaluation Method | DS-STAR+ Result |
|-----------|-----------------|-----------------|
| DABStep-Research | LLM-as-judge (Gemini-3-Pro + GPT-5), quality score 1–5 on content and formatting | **88%+ win rate** over DeepAnalyze-8B (best baseline) |
| DABStep-Research categories covered | Data preparation, data analysis, data insight, report generation, open-ended research | State-of-the-art across all 5 |

### Ablation Study Key Findings

| What was removed | Impact |
|-----------------|--------|
| Analyzer | Hard task accuracy: 45.2% → **26.98%** (−18 pp). Most critical component. |
| Router (backtracking disabled) | Measurable accuracy drop. Backtracking is essential for correcting plan-level errors. |
| Iterative verification (single-pass only) | Accuracy drops; the loop is necessary for multi-step problems. |

## Part 4: Key Differentiators vs Prior Systems

1. **Heterogeneous file format support**: Not limited to SQL or structured databases. CSV, JSON, Markdown, unstructured text — all in one query over mixed files.

2. **One-step-at-a-time iterative planning**: Prior systems generate a complete plan upfront. DS-STAR generates one step at a time, conditioning each new step on the actual execution result of the previous one. Plans adapt to observed data reality.

3. **Verification-driven loop**: The system does not terminate after a fixed number of rounds. It terminates when the Verifier judges the result sufficient, up to a maximum of 20 rounds. Quality determines termination, not iteration count.

4. **Backtracking Router**: When a plan goes wrong, the Router can identify the erroneous step and truncate back to it — not just append more steps. This prevents error propagation where all subsequent steps build on a flawed foundation.

5. **Open-ended report generation** (DS-STAR+): Not just single-answer QA. DS-STAR+ can produce multi-section research reports with citations, covering a topic from multiple analytical angles.

6. **Cited, auditable outputs**: Every claim in a DS-STAR+ report traces back to a sub-question, a Python script, and the raw data. The audit trail is built in.

---

## Part 5: Limitations (from paper)

1. **Computational cost**: Running DS-STAR+ with n sub-questions means n full DS-STAR executions, each up to 20 rounds. For complex open-ended queries, total compute time can be significant.

2. **Code execution scope**: The system can only answer questions answerable through Python computation on available files. Questions requiring domain knowledge not present in the data cannot be answered.

3. **Maximum round limit**: Setting max rounds too low can result in premature best-effort answers. Setting too high increases cost.

4. **LLM quality dependency**: The Verifier and Router are themselves LLM calls. If those models make incorrect judgments, the loop may terminate early or backtrack unnecessarily.

5. **Data quality sensitivity**: The Analyzer's descriptions are only as good as the data. Poorly labelled columns or inconsistent data encoding can mislead the Planner.
