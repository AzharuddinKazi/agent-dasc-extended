🏗️  Production Hardening — What v2 Missed (Architect Review)
🔄 HA at Every Critical Node
v2 had single instances everywhere — one crash kills the whole system.

Fixed in v3: FastAPI (2 replicas) · Redis Sentinel (3-node) · PostgreSQL (primary + streaming standby) · LLM API endpoints (primary + failover region per tier) · MinIO (erasure-coded multi-node).
✅ No single point of failure in critical path
💾 Task Recovery via Checkpointing
v2 had no recovery path. A Celery worker crash on round 12 of 20 = task lost, analyst re-submits from scratch.

Fixed in v3: LangGraph checkpoints state to PostgreSQL after every round. On worker death, a new worker picks up the task from last checkpoint and continues. Zero task loss on deploy or crash.
✅ Tasks survive worker restarts and rolling deploys
📋 Priority Queue + Dead Letter Queue
v2 had a flat task queue. An Enforcement case query could wait behind 20 research queries.

Fixed in v3: Three priority queues (CRITICAL · HIGH · NORMAL). Enforcement queries go to CRITICAL, jump ahead. Failed tasks land in DLQ — analyst and ops team notified with reason. No silent failures.
✅ Time-sensitive investigations never blocked by routine work
🔑 Secrets Management
v2 had no secrets strategy — credentials would have lived in env vars or config files. In a multi-tenant SaaS system, this is a critical risk.

Fixed in v3: a cloud secrets manager (e.g., AWS Secrets Manager, GCP Secret Manager). All DB credentials, LLM API keys, and integration tokens fetched at runtime and auto-rotated. No secret touches a config file or container image.
✅ Zero hardcoded secrets anywhere in the stack
📂 Data Versioning for Reproductibility
v2 showed data volumes with no versioning. Source data is refreshed periodically. If an analyst finds an issue, they must be able to reproduce the exact result months later (for follow-up analysis or a business review).

Fixed in v3: Data Version Manager creates immutable snapshots on each ingest. Every query result is linked to the exact data version used. Point-in-time replay guaranteed.
✅ Every finding is reproducible — legally defensible
🔒 mTLS + Network Isolation
v2 had no internal transport security. Internal service traffic was plaintext, which is a gap even for cloud-internal traffic.

Fixed in v3: mTLS enforced on all service-to-service connections (FastAPI ↔ Celery, LangGraph ↔ LLM, Executor ↔ Container daemon). Containers have --network=none — zero network egress. Sandbox stdout capped at 10KB to prevent data exfiltration via print flooding.
✅ Defense in depth — no plaintext internal traffic
📊 LLM Inference HA + Queue
v2 assumed a single LLM endpoint. 10 concurrent DS-STAR+ tasks × 6 sub-questions = 60 simultaneous LLM requests hitting one region. This causes rate-limiting and cascading failures.

Fixed in v3: Request router across primary and failover regions for both Flash and Pro tiers. Internal LLM inference queue serializes concurrent requests. Failover region promoted automatically if primary is degraded.
✅ LLM layer can handle burst load without dropping requests
📦 Container Lifecycle: Task-Scoped, Not Round-Scoped
v2 showed containers as ephemeral per round. With DS-STAR+ (8 sub-questions × 20 rounds), that would be 160 container cold-starts per query (~3s each = 8 minutes of pure overhead). Unacceptable for analyst UX.

Fixed in v3: One container per task, alive for all rounds. Each round runs its complete sₖ script via docker exec. The Coder builds sₖ incrementally (extending sₖ₋₁) but always outputs a fully self-contained runnable script — it re-imports and re-computes from scratch each round, making stale container state a non-issue. Rounds get warm execution with zero cold-start penalty.

DS-STAR+: one container per sub-question (sub-questions share no state). Max 5 containers concurrent system-wide. Sequential sub-question execution by default; parallel mode available via config flag.
✅ Container sprawl eliminated · Cold-start overhead eliminated · Predictable resource usage
🎯 Tamper-Evident Audit Log
v2 used MinIO for audit logs ("append-only by convention"). Audit logs should be tamper-evident so that any modification is detectable.

Fixed in v3: Append-only PostgreSQL table with write-lock enforcement. SHA-256 hash chain — each row hashes the previous row. Any row modification or deletion is immediately detectable. 10-year retention with separate backup schedule.
✅ Audit log supports internal and external compliance review
🛑 Graceful Shutdown
v2 had no shutdown strategy. A rolling deploy or restart would kill in-flight agent tasks mid-round, corrupting LangGraph state and losing results.

Fixed in v3: LangGraph checkpoints before shutdown signal. Celery workers drain existing tasks before accepting SIGTERM. New version starts only after all in-flight tasks complete or are checkpointed. Zero downtime deploys possible.
✅ Deploys are safe — no analyst query is ever lost
⚡ PII Masking Order
v2 ran Explainer before PII Masker. This meant the LLM generating the plain English narrative could see raw PII (account numbers, national IDs, names) and might include them in the explanation — then masking would need to parse free text, which is unreliable.

Fixed in v3: PII Masker always runs FIRST on structured data (DataFrames, JSON), then the Explainer receives only masked data. Structural masking on structured output is deterministic and complete.
✅ PII never reaches the Explainer LLM — deterministic protection
📬 Config / Prompt Versioning
v2 had no config or prompt management. Domain knowledge bases, prompt templates, and metric-definition libraries had no home — they would have been hardcoded in agent code, making updates require a full code deploy.

Fixed in v3: Dedicated Config & Prompt Store, version-controlled and separate from code. Prompt updates (e.g., adding a new metric definition or business taxonomy entry) are configuration changes, not deployments. Rollback on regression.
✅ Domain knowledge updates don't require code deployments
🔔 Async Notification + WebSocket Resilience
v2 assumed WebSocket stays alive for 10–30 minute tasks. In practice, browser tabs are closed, sessions time out, and proxies drop long-lived connections.

Fixed in v3: Notification Service sends in-app alerts when tasks complete. WebSocket client auto-reconnects and reattaches to in-flight task by task_id. 30s ping keepalive prevents proxy timeout. Analysts don't need to stay on the page.
✅ Task completion never missed even if browser closes
